## Задача — Создать архитектурное решение для аналитического пайплайна ##

# Архитектурное решение:

- Для данной задачи была выбрана классическая трехслойная архитектура проектирования `DWH: Raw → Core → Marts`.
- Стек: (`Airflow` - как оркестратор) --> (`ClickHouse` - как хранилище и формальный OLTP слой) --> (`Superset` - как визуализатор).
- Схема лямбда-архитектуры: S3 (`lab08_daily_dag.py` грузит батч каждые 10 мин) ──→  хранилище (`lab08.transactions_raw`) ──→  дашборды
Kafka (`lab08_daily_dag.py `грузит стрим, real-time в lab08.transactions_raw в 4 потока)↗  

# Техническое описание:

- Загружает батч-данные из S3 (Yandex Cloud) через Airflow:

    * `lab08_daily` — разово загружает справочники в таблицы: (users, test_users, promo_codes), и ежедневно отмены транзакций(cancellations_raw) и курсы валют(exchange_rates_raw);
    * `lab08_transactions` — каждые 10 минут грузит слот транзакций из `s3://npl-de18-lab8-data/day=.../slot=.../transactions.jsonl`;

- Получает стриминг-данные через Kafka --> ClickHouse через Kafka Engine:

    * Один топик `lab08_transactions` содержит три типа событий: транзакции, отмены, курсы валют;
    * Данные грузятся в таблицу `lab08.kafka_events`;
    * Materialized Views раскладывают события по соответствующим raw-таблицам;

- ClickHouse выступает в роли DWH: получение данных, очистка, создание витрин;

- Описание объектов БД (`lab08.*`):

    * `kafka_events` — принимает данные в формате `JSONEachRow` из топика `lab08_transactions`;
    * `kafka_transactions_mv` — Мат-вьюха которая матереализуеют данные в `transactions_raw`;
    * `kafka_cancellations_mv` — Мат-вьюха которая матереализуеют данные в `cancellations_raw`; 
    * `kafka_rates_mv` — Мат-вьюха которая матереализуеют данные в `exchange_rates_raw`;
    * `transactions_raw` — все транзакции (батч + стрим), дедупликация через ReplacingMergeTree;
    * `transactions_clean` — "OLTP слой";
    * `cancellations_raw`, `exchange_rates_raw` — отмены и курсы;
    * `users`, `test_users`, `promo_codes` — справочники;
    * `users_dict` — словарь ClickHouse для быстрого лукапа is_test_user вместо JOIN;

- Визуализируем данные в Apache Superset (Общий дашборд `Lab08 Аналитика` из `superset_exports/`).

- Аналитические витрины:

  * `mart_hourly_distribution` — распределение транзакций по часам;
  * `mart_purchases_hourly` — покупки по часам;
  * `mart_revenue_daily` — выручка в TGRK по дням;
  * `mart_promo_analysis` — использование промокодов;
  * `mart_cancellations` — отмены по причинам и датам, среднее время до отмены;
  * `mart_test_vs_real` — сравнение тестовых и реальных пользователей по типам транзакций;

 # Бизнес логика:

- Бизнес хочет видеть "инсайты" в транзакции пользователей: сколько денег проходит через систему, какие промокоды работают, где отмены, как себя ведут тестовые пользователи и вообще можно ли доверять этим данным.

- Данные поступают из двух источников одновременно:

    * Батч-данные на S3 — каждые 10 минут появляется новый файл с транзакциями за последний интервал;
    * Стрим событий в Kafka — те же данные, но в реальном времени, событие за событием;

- Нужно построить полноценный аналитический пайплайн, который забирает данные из обоих источников, складывает в подходящее хранилище, считает метрики и показывает дашборд с визуализациями.

- Данные поступают непрерывно в течение примерно месяца (около 30 дней), начиная с 27 апреля с возможно некоторым запаздыванием. Данные грязные. Вот что может встретиться:

    Пустой user_id — ~5% транзакций без привязки к пользователю
    Несуществующие user_id — ~3% транзакций ссылаются на пользователей, которых нет в справочнике
    Отрицательные суммы — ~10% (это не баг — но формально поле amount может быть < 0)
    Дублирующиеся transaction_id — ~5% (одинаковые ID у разных транзакций)
    Нулевые суммы — ~1% транзакций с amount = 0
    Использование просроченных промокодов — ~2%
    Тестовые пользователи — ~20% трафика в рабочие дни, 0% в выходные. Нужно ли их учитывать?
    Разные форматы дат — created_at в unix timestamp, cancelled_at в строковом формате без таймзоны

- Бизнес требования для витрин:

    Распределение транзакций по часам — столбики.
    Количество покупок по часам — таблица или пироги. 
    Выручка в базовой валюте — конвертация через курсы валют, агрегация по дням.
    Анализ промокодов — какие работают, использование vs лимиты, просроченные.
    Анализ по источникам трафика (тестовые vs реальные пользователи).
    Отмены: процент, причины, среднее время до отмены.
    Когортный анализ пользователей (новые vs возвращающиеся).

# Как запустить?

1. Клонируем репозиторий.
```
git clone <repo-url>
cd lab08
```

2. Создаём `.env` из шаблона и заполняем своими значениями.
```
cp .env.example .env
```
Обязательно заполнить:
```
POSTGRES_PASSWORD=your_secure_password
AIRFLOW_ADMIN_PASSWORD=your_admin_password
SUPERSET_ADMIN_PASSWORD=your_admin_password
SUPERSET_SECRET_KEY=random_secret_key
SLACK_WEBHOOK_URL=https://hooks.slack.com/...   # опционально
```

3. Копируем шаблон Kafka и прописываем реальный адрес брокера.
```
cp init/KAFKA_DATA_GIT.sql init/KAFKA_DATA.sql
```
Заменяем `your-kafka-host:9092` на реальный адрес в строке `kafka_broker_list`.

4. Запускаем.
```
docker compose up -d
```
`airflow-init` автоматически мигрирует БД, создаёт пользователя, добавляет Connection и анпаузит DAG-и.

# Создание пользователя

Пользователь `admin` создаётся автоматически при первом запуске через `airflow-init`.
Логин и пароль — из переменных `AIRFLOW_ADMIN_USER` / `AIRFLOW_ADMIN_PASSWORD` в `.env`.
Аналогично для Superset: `SUPERSET_ADMIN_USER` / `SUPERSET_ADMIN_PASSWORD`.
_________________________________________________________________________________________________________________________________________

# Запуск DAG

DAG-и запускаются автоматически (catchup=True, start_date=2026-04-27).

Чтобы запустить вручную в Airflow UI:
1. Открываем http://localhost:8080
2. DAG `lab08_daily` → Trigger DAG
3. DAG `lab08_transactions` → Trigger DAG

Проверка данных в ClickHouse:
```sql
SELECT count() FROM lab08.transactions_raw;
```
